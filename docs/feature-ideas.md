## 1. Что уже круто

Проект на удивление зрелый для MVP. Вот что реально хорошо:

- **145 полей в 24 секциях** — глубоко проработанная модель персонажа. Не поверхностный генератор, а инструмент для серьёзных авторов.
- **AI-анализ** (contradictions, clichés, gaps, inconsistencies, opportunities) — это единственное, что отличает редактор от просто текстовой формы. Убийственная фича для сценаристов.
- **Word-level diff** перед применением AI-изменений — пользователь не теряет контроль.
- **6 AI-провайдеров** с каскадным UI-селектором — редкость даже в коммерческих продуктах.
- **Staged/Apply паттерн** для AI-настроек — изменения не вступают в силу без «Применить».
- **SSE-стриминг** с парсингом partial JSON — пользователь видит прогресс.
- **История анализов** с кэшированием (localStorage) — не дёргает API повторно.
- **Редактор системных промптов** — power-user фича, позволяет тонко настраивать поведение AI.
- **World elements** (локации, фракции, история, законы) — зачаток вики-подсистемы.
- **Auth через NextAuth** с авторегистрацией — идеально для персонального инструмента.
- **API-ключи в HttpOnly cookies**, а не в localStorage — правильно с точки зрения безопасности.

## 2. Технический долг и узкие места

Это не «сломано», а скорее «сделано на скорую руку» и будет мешать при добавлении функций.

### 🔴 Блокирующие рост

| Проблема | Почему важно | Что делать |
|----------|-------------|------------|
| **145 полей в одной JSON-колонке** | Нельзя индексировать, искать, фильтровать по конкретным полям. Любой поиск = парсинг JSON всего файла БД. При добавлении связей между персонажами станет больно. | Мигрировать на EAV или документную БД (SQLite с generated columns, или PostgreSQL с JSONB). Минимально: добавить generated columns для ключевых полей (name, gender, archetype). |
| **Rate limiter in-memory** | Сбрасывается при рестарте. При масштабировании за контейнером — гонка. | SQLite-таблица `RateLimit` с TTL-очисткой по крону. |
| **Дублирование SSE-логики** | `fill/route.ts` и `analyze/route.ts` содержат идентичную логику стриминга, хотя частично вынесена в `streamUtils.ts`. В хуках (`useAiFill`, `useCharacterAnalysis`) — ещё один слой дублирования чтения SSE. | Добить абстракцию: `fetchSseStream()` в хуках уже есть в `prompt-parser.ts`, но НЕ используется в `useAiFill.ts` и `useCharacterAnalysis.ts`. |
| **Дублирование PROVIDER_MODELS** | Модели прописаны в ДВУХ файлах: `useAiSettings.ts` (клиент) и `provider.ts` (сервер). Добавление модели = правка в двух местах. | Единый source of truth: конфиг на сервере → API-роут `GET /api/ai/models` → клиент получает список оттуда. |

### 🟡 Умеренные

| Проблема | Что делать |
|----------|-----------|
| `CharacterForm.tsx` (586 строк, 25 KB) | Вынести: автосохранение → хук `useAutoSave`, undo/redo → хук `useUndo`, keyboard shortcuts → хук `useKeyboard`. |
| `DashboardClient.tsx` (612 строк, 29 KB) | Разбить на `ProjectHeader`, `WorldElementsPanel`, `CharactersGrid`. |
| Анализы в localStorage | Нет синхронизации между устройствами. При большом количестве — забивает localStorage (лимит ~5 MB). Мигрировать в БД (новая модель `AnalysisRecord`). |
| `AppSetting` — заготовка | Модель есть в Prisma, но используется только для промптов. Можно расширить: темы, настройки редактора, избранные модели. |
| xAI и OpenAI не тестировались | Код написан, но реальных ключей в `.env` нет — неизвестно, работают ли. |

### 🟢 Некритичные

- Нет CORS на API-роутах (для локального использования норм).
- `prisma migrate deploy` вместо `db push` в Docker — docker-entrypoint.sh использует `db push --accept-data-loss`.
- Не везде консистентный обработчик ошибок AI — часть роутов использует `routeUtils.ts`, часть нет.

## 3. Идеи новых функций

### 3.1 Связи и граф персонажей

**Что:** персонажи в проекте могут быть связаны друг с другом (родство, дружба, вражда, романтика, подчинение). Визуальный граф + описание отношений.

**Почему:** сейчас персонажи — изолированные острова. Сценаристу нужно видеть сетку отношений.

**Реализация:**
- Новая модель `CharacterRelation`: `sourceId`, `targetId`, `type` (enum), `description`, `strength` (1-5).
- Компонент `RelationGraph` — визуализация на Canvas/SVG с draggable nodes.
- AI-генерация отношений: «Проанализируй персонажей проекта и предложи возможные связи».
- В карточке персонажа — блок «Связи» с мини-графом.

**Сложность:** средняя (новая модель + миграция + визуализация).

---

### 3.2 Таймлайн и бэкстори

**Что:** хронологическая лента событий жизни персонажа. Возрастные вехи, ключевые события, травмы, поворотные моменты.

**Почему:** 145 полей описывают статический срез. Сценаристу нужно понимать, *как* персонаж стал таким.

**Реализация:**
- Новая модель `TimelineEvent`: `characterId`, `age` (или `date`), `title`, `description`, `impact` (1-5).
- Компонент `TimelineView` — вертикальная лента с маркерами возраста.
- AI-генерация: «По заполненным полям предложи 5-10 ключевых событий из прошлого персонажа».
- Визуализация «до/после»: два снапшота персонажа в разном возрасте.

**Сложность:** средняя.

---

### 3.3 Генератор сцен и диалогов

**Что:** на основе персонажей проекта AI генерирует сцены: «Что будет, если Алексей встретит Марину в заброшенной библиотеке?»

**Почему:** самое интересное в персонажах — их взаимодействие. Сейчас можно только заполнять поля.

**Реализация:**
- API-роут `/api/ai/scene` — принимает список `characterIds`, `location`, `premise`.
- Генерирует: описание сцены, диалог, внутренние монологи персонажей.
- Компонент `SceneGenerator` — выбор участников + контекст + результат в формате «сценарий».
- Дополнительно: AI генерирует *только* диалог между двумя персонажами в их речевых паттернах.

**Сложность:** средняя (новый роут + компонент + промпт).

---

### 3.4 Сравнение и архетипы

**Что:** side-by-side сравнение двух персонажей + автоматическая классификация по архетипам (Кэмпбелл, Юнг, Пропп).

**Реализация:**
- Компонент `CompareView` — две колонки, подсветка различий.
- AI-фича: «Найди общие черты и ключевые различия между А и Б».
- AI-фича: «К какому архетипу ближе этот персонаж?» (Герой, Трикстер, Ментор, Тень, etc.)
- Статистика по проекту: сколько персонажей каждого архетипа, какого пола, возраста.

**Сложность:** низкая-средняя.

---

### 3.5 Речевые паттерны и голос

**Что:** новые поля в схеме персонажа для речевых характеристик: темп речи, слова-паразиты, акцент, синтаксис, любимые выражения. AI генерирует образцы речи.

**Почему:** «голос» персонажа — критичен для диалогов, но в текущей схеме его нет.

**Реализация:**
- Новая секция `voice` в `CHARACTER_SCHEMA` (~6-8 полей).
- AI-фича: «Сгенерируй 5 фраз, которые этот персонаж мог бы сказать».
- При генерации сцен (см. 3.3) AI учитывает речевые паттерны.

**Сложность:** низкая (добавить секцию в schema.ts + AI-промпт).

---

### 3.6 Арка персонажа

**Что:** отслеживание изменений персонажа по ходу истории. От начального состояния → через кризис → к трансформации.

**Почему:** сценаристу нужно видеть траекторию развития, а не только статический портрет.

**Реализация:**
- Новая модель `CharacterSnapshot`: версия данных персонажа на определённом этапе.
- Компонент `ArcView` — ползунок «этап 1 → этап 2 → этап 3», сравнение версий через DiffModal.
- AI-фича: «Предложи арку для этого персонажа в трёх актах».

**Сложность:** высокая (версионирование данных + UI).

---

### 3.7 Импорт / Экспорт

**Что уже есть:** экспорт в Markdown, JSON, Plain Text.
**Чего не хватает:**

- **Импорт из JSON/Markdown** — обратная операция. Распарсить файл → создать нового персонажа. Уже в roadmap (этап 5.3).
- **Пакетный экспорт** — все персонажи проекта в один файл.
- **Экспорт в PDF** — красивая вёрстка карточки для печати/отправки.
- **Экспорт в Obsidian** — Markdown с YAML frontmatter (тэги, свойства в `---` блоке).
- **Экспорт в формат SillyTavern / Character.AI** — JSON-схема совместимая с популярными AI-чатами.
- **Drag-and-drop импорт** — бросил файл на дашборд → персонаж создан.

**Сложность:** низкая-средняя.

### 3.8 AI: голосование моделей и стилизация

**Что:** продвинутые AI-фичи, использующие мультипровайдерную архитектуру.

**Multi-model voting:**
- Один запрос → 2-3 модели → сравнить результаты → показать лучший (или мерж).
- «DeepSeek заполнил поле зло, Grok — мягче, выбери вариант».
- Минимизирует риск галлюцинаций одной модели.

**Style transfer:**
- «Перепиши поля этого персонажа в стиле Достоевского / Пратчетта / Кинга».
- AI получает стилевой референс (отрывок текста) и адаптирует описания.

**Image-to-character:**
- Загрузил портрет → AI (Gemini Vision / Claude Vision) описывает внешность и характер.
- Предзаполняет секции «Внешность» и «Характер».

**Voice-to-character:**
- Надиктовал описание голосом → Whisper транскрибирует → AI заполняет поля.

**Сложность:** средняя-высокая.

### 3.10 Проект как вики

**Что:** проект — не просто папка для персонажей, а полноценное «мировое досье».

**Реализация:**
- **World elements** уже есть (локации, фракции, история, законы) — расширить: артефакты, магические системы, технологии, календарь.
- **Project notes** — свободные заметки с Markdown.
- **Глоссарий** — термины мира с перекрёстными ссылками.
- **AI-генерация лора** — «Опиши магическую систему этого мира на основе созданных персонажей».
- **Связи world elements с персонажами** — «Этот персонаж из фракции X», «живёт в локации Y».

**Сложность:** средняя-высокая.

### 3.11 Telegram-бот и API

**Что:** возможность работать с персонажами через Telegram — быстро добавить заметку, получить сводку.

**Реализация:**
- Отдельный Telegram-бот (в стиле Hermes-профилей).
- Команды: `/char <имя>` — сводка персонажа, `/note <текст>` — добавить заметку к последнему.
- Голосовые сообщения → Whisper → автосохранение в редактор.
- REST API для внешних интеграций (GET/POST персонажей).

**Интеграция с Hermes:** интересный вариант — Hermes-профиль, который через API дёргает Character Editor как backend. Пользователь в Telegram: «Опиши главного злодея для моего детектива» → Hermes → Character Editor API → создан персонаж.

**Сложность:** средняя.

---

### 3.12 Совместная работа

**Что:** несколько авторов работают над одним проектом.

**Реализация:**
- **Sharing по ссылке** — read-only доступ к персонажу/проекту.
- **Комментарии** к конкретным полям — «Этот мотив не стыкуется с прошлым».
- **Версионирование** — история изменений каждого поля.
- Сложность статуса: «черновик» / «на ревью» / «готов».

**Сложность:** высокая (требует переработки модели прав, WebSocket для real-time).

### 3.13 Визуализация и изображения

**Что:** сейчас проект текстоцентричный. Добавить визуальный слой.

**Реализация:**

**Сложность:** средняя (интеграция image-gen API).

## 4. Приоритизация: quick wins vs big bets

### 🟢 Quick Wins (1-3 часа каждая, можно сделать сегодня)

1. **Индикатор автосохранения** — зелёная галочка «Сохранено» / спиннер «Сохраняю...»
2. **Горячие клавиши** — Ctrl+S, Ctrl+Z, Ctrl+Shift+Z
3. **Прогресс-бар AI-заполнения** — X/Y секций, X/Y полей
4. **Тёмная тема без перезагрузки** — CSS-переменные
5. **Секция «Речевые паттерны»** — 6-8 полей в schema.ts + AI-промпт для генерации фраз
6. **Импорт из JSON** — парсинг + createCharacter
7. **Source of truth для моделей** — `GET /api/ai/models` → клиент получает список

### 🟡 Strategic Bets (1-3 дня каждая, меняют продукт)

8. **Связи и граф персонажей** — модель `CharacterRelation` + визуализация
9. **Таймлайн и бэкстори** — модель `TimelineEvent` + AI-генерация событий
10. **Генератор сцен и диалогов** — `/api/ai/scene`
11. **Сравнение персонажей + архетипы** — `CompareView` + AI-классификация
12. **Экспорт в Obsidian / Character.AI форматы**

### 🔴 Big Bets (неделя+, архитектурные изменения)

13. **JSON-колонка → нормализованные данные** — миграция на generated columns или PostgreSQL
14. **Арка персонажа** — версионирование данных
15. **Совместная работа** — sharing, комментарии, real-time
16. **Telegram-бот как интерфейс к редактору**
